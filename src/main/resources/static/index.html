工程阅读器：先把项目跑起来，再让 AI 帮你读

当前页面已经不是纯展示稿。它会在本地后端启动后调用接口，用 mock AI 生成第一版结构化阅读报告。

先知道它做什么 它帮助团队理解系统运行状态，不是普通业务应用。
再知道数据怎么流 数据从 agent 采集，进入 server，再被查询和展示。
最后看怎么运行 先跑官方 demo 或最小部署，再回头看源码。

怎么最快跑起来

  1. 启动 Spring Boot 后端。
  2. 打开这个 H5 页面。
  3. 输入本地项目路径，点分析。
  4. 看报告是否能指导下一步阅读。
mvn spring-boot:run

端到端链路

1
Agent 采集数据 部署在节点上，采集网络、系统调用、应用和资源数据。
2
数据进入后端 Server 接收和处理采集数据，做清洗、聚合和存储。
3
查询和分析 用户通过 UI、API 或查询层查看链路、指标和事件。
4
定位问题 最终目标是回答“哪里慢、哪里断、谁影响了谁”。

第一次应该读哪些文件

  • README.md：确认项目定位、运行方式和官方推荐路径。
  • pom.xml：确认后端依赖和 Spring AI 接入方式。
  • src/main/java：沿 Controller、Service、AiGateway 看主链路。

30 天计划：每天都有一个可见产出

这个计划不是为了每天逼自己学很多，而是为了每天留下一点能被别人看到的证据：代码、页面、文档、截图、公开记录都算。

第 1 周：让 demo 真能用

  1. 定主线并发第 1 篇：为什么做 AI 工程阅读器。
  2. 重构四件套模型：项目概览、运行方式、核心链路、关键文件。
  3. 升级本地扫描器：README、pom、启动类、Controller、Service、Gateway。
  4. 页面展示四件套：让 tab 内容来自接口结果。
  5. 核心链路第一版：识别 Controller -> Service -> AiGateway。
  6. 自举 demo：用它分析它自己。
  7. 第一周复盘：README、截图、作品集日志。

第 2 周：让 AI 输出更像工程系统

  1. Spring AI 真实调用。
  2. 结构化输出稳定性和兜底。
  3. PromptBuilder 分层管理。
  4. 带证据文件的追问问答。
  5. 扫描缓存和分析结果缓存。
  6. Markdown 导出报告。
  7. 可展示 demo v0.2 和面试讲述稿。

第 3 周：让它读真实工程

  1. 选择一个真实业务项目或开源 Spring Boot 项目。
  2. 支持更复杂目录和无关文件过滤。
  3. 优化 Top 5 文件排序规则。
  4. 提取运行方式和依赖。
  5. 支持围绕链路继续追问。
  6. 记录一次 AI 分析失败样本。
  7. 形成真实项目样例报告。

第 4 周：作品集化和求职表达

  1. README 改成作品集可读版本。
  2. 补架构图。
  3. 整理技术亮点。
  4. 写 3 分钟演示脚本。
  5. 录屏或准备可访问 demo。
  6. 补测试。
  7. 整理面试 Q&A 和 30 天总复盘。

公开发布规则

  • 工作日最低 45 分钟，必须留下一个小产出。
  • 周末用 4-6 小时做大块功能和复盘。
  • 每天公开记录只讲一个点，不写成论文。
  • 先发草稿，最后发布由你确认。
