# Day 1 想法卡：代码热读工具

日期：2026-07-19

## 一句话目标

做一个 AI 驱动的代码热读工具，帮助用户快速理解一个工程的主线、用途、运行方式和核心链路，并尽快跑出 demo。

## 我想做什么

我想做一个工具，能快速帮人理清楚一个工程：

- 它到底是做什么的
- 它解决什么问题
- 它需要什么配置
- 它怎么部署
- 它依赖什么
- 它用到哪些资源
- 如果我要基于它做建设，需要掌握哪些技术
- 这些技术应该怎么快速入门
- 能不能帮用户尽快跑一个 demo 出来

本质上，我想做的不是“代码总结器”，而是一个 **工程热读助手**。

它要帮助用户在信息密度很高、变化很快的时代，快速抓住一个项目的精髓。

## 这个工具解决谁的问题

### 1. 像我这样容易想深、想重的人

痛点：

- 容易一上来就陷入大量细节
- 容易害怕自己不懂
- 容易在开始之前就被复杂度压住
- 容易不知道主线是什么
- 容易忘记目标其实是先跑通一个 demo

这个工具要帮助这类人：

- 快速抓主线
- 快速抓主要矛盾
- 先跑 demo
- 再逐步深入

### 2. 对新技术恐惧的小白用户

痛点：

- 不知道一个技术项目从哪里开始读
- 不知道要先配什么环境
- 不知道哪些依赖是关键的
- 不知道跑 demo 的最短路径
- 看到代码和文档就容易放弃

这个工具要帮助这类人：

- 先知道项目在干什么
- 再知道怎么跑起来
- 再知道关键链路在哪里
- 最后知道如果要继续学，下一步学什么

## 用户可能输入什么

第一版可以先支持最简单的输入：

- 本地项目路径
- README
- 项目目录树
- `pom.xml` / `package.json` / `build.gradle` 等依赖文件
- 用户当前目标，例如“我想先跑通 demo”或“我想理解主链路”

后续可以扩展：

- GitHub 仓库地址
- 用户提问
- 用户选择的阅读模式
- 用户已有背景，例如 Java 后端、数据工程、小白、面试准备

## 系统应该输出什么

第一版输出不追求完整，只追求能让用户行动。

候选输出：

- 项目一句话说明
- 这个项目解决的问题
- 适合先跑的 demo 路径
- 需要准备的环境和配置
- 核心依赖和它们的作用
- 关键入口文件
- 主链路说明
- 推荐先读的 3-5 个文件
- 每个文件为什么重要
- 如果要基于这个项目做建设，需要补哪些技术
- 这些技术的最快入门顺序
- 当前不确定点
- 下一步行动清单

## 30 天目标

30 天内做出一个可展示的 AI 工程热读助手。

### 第 1 周

做出一个小版本：

- 能读一个简单项目
- 能输出项目阅读地图
- 能告诉用户先看什么、怎么跑 demo

### 第 2 周

做出一个基本可用的 MVP：

- 支持结构化输出
- 支持更清楚的阅读步骤
- 能保存输入输出样本
- 能对输出做简单人工验收

### 第 3-4 周

让它更通用一点：

- 支持不同类型项目
- 能识别核心资源和关键依赖
- 能对某条核心链路做更细的解释
- 尝试对关键资源做 mock 或 demo 化说明

## 例子：如果读 Spring 源码

理想情况下，工具不仅告诉用户“Spring 是什么”，还要帮用户看见核心链路：

- 用户输入从哪里进来
- 框架怎么拿到底层资源
- 配置怎么生效
- 工厂/容器/生命周期大概怎么串起来
- 如果想 mock 一个最小链路，应该 mock 哪些核心资源

目标是让用户一目了然地 get 到主链路，而不是一开始就淹没在源码细节里。

## 强制要求

这个项目必须用到至少一个 AI 相关框架或工程能力。

候选方向：

- Spring AI
- LangChain
- LlamaIndex
- OpenAI SDK / Agents SDK
- 其他 AI Studio / agent / workflow 框架

原因：

- 这是作品集项目
- 需要体现我在 AI 时代的工程应用能力
- 我需要真正掌握至少一种 AI 应用框架

## 先不做什么

第一阶段不做：

- 不做完整商业产品
- 不做复杂前端
- 不做训练模型
- 不做 fine-tuning
- 不做支持所有语言和所有项目
- 不做一上来就很复杂的 agent
- 不追求一次读完整个大项目

第一阶段只做：

- 选一个简单项目
- 读出主线
- 输出项目阅读地图
- 帮用户跑 demo
- 记录输入输出
- 为后续迭代留下样本

## 当前最重要的问题

下一步要和 Codex 讨论清楚：

- 用户输入的最小集合是什么
- 系统输出的最小结构是什么
- 第一版到底选哪个项目作为样本
- 第一版是否先用当前项目自举
- AI 框架第一阶段选 Spring AI 还是先保留 mock，再在第 2 周接入

## 参考字帖：Codebase Explainer

链接：https://www.ashishmehrotra.com/projects/codebase-explainer

浏览器收藏：

- 已加入 Chrome 收藏夹
- 以后可以在浏览器收藏里搜索：`Codebase Explainer`
- 如果忘记名字，也可以搜索关键词：`codebase` / `explainer` / `Top 5 Files`

在本项目里怎么找：

- 打开 `docs/08-day1-idea-card.md`
- 搜索：`Codebase Explainer`
- 它是本项目 Day 1 的第一个竞品字帖

这个链接的用途：

- 用来看“类似产品的最小输出结构”
- 用来看“别人怎么把代码库阅读包装成新人指南”
- 用来提醒我第一版不要贪大

当前不要用它做什么：

- 不要现在研究它完整技术架构
- 不要现在复刻它的多个 agent 方案
- 不要因为它提到 Google ADK、Crew AI、LangGraph 就跑偏

这次只带四个问题去看：

- 它是干什么的？
- 它的输入是什么？
- 它的输出是什么？
- 它和我现在要做的东西契不契合？

我的理解：

- 它是一个 codebase explainer，可以读一个代码库并生成解析结果
- 它的输入很轻，只需要一个 GitHub URL
- 它的输出包括：
  - New Developer Guide / 新人指南
  - Data Flow / 数据流转或主链路
  - Top 5 Important Files / 五个重要文件
- 它从全局上给指南，从时间/流程上给数据流转，从重点上给 Top 5 文件
- 这个输出结构和我的目标非常契合

对我的项目的启发：

- 第一版也可以先做三件事：
  - 新人上手指南
  - 主链路说明
  - Top 5 文件
- 后续再加：
  - 最短 demo 路径
  - 下一步学习建议
  - 不确定点

暂时不深入的部分：

- 它提到的 Google ADK、Crew AI 等框架
- 它为什么用了多个 agent
- 具体实现细节

后续可以再看的线索：

- LangGraph：它可能适合表达“循环执行、决策、下一步动作”的流程
- 等我项目最小版本跑通后，再回来研究它的框架设计

当前标记：

**这个项目是输出结构参考，不是今天要深入复刻的技术方案。**
