# 产品定义 v2：低门槛工程热读工作台

日期：2026-07-19

## 一句话定位

这是一个基于 Spring AI 的工程热读工作台。

用户不需要自己整理 README、目录树和源码片段，只要提供一个低门槛入口，系统就自动扫描项目，生成项目地图、核心链路、Top 5 文件和阅读报告。

## 为什么不是裸模型

裸模型也能回答代码问题，但它要求用户自己做很多前置工作：

- 自己找 README
- 自己复制目录树
- 自己判断哪些文件重要
- 自己组织问题
- 自己判断回答有没有编造

本产品的价值是把“读项目”做成固定工作流：

```text
低门槛输入
-> 自动扫描项目
-> 自动选择关键上下文
-> 生成固定结构输出
-> 带文件证据
-> 缓存项目上下文
-> 支持持续追问
```

## 输入：只支持两种低门槛方式

### 1. GitHub URL

用户输入一个 GitHub 仓库地址。

系统负责：

- 拉取或读取仓库信息
- 识别 README
- 识别构建文件
- 识别目录结构
- 识别关键源码文件
- 生成项目上下文

用户不需要手动复制项目资料。

### 2. 集成到用户本地项目

用户把工具下载到本地，或者集成到自己的项目里。

系统负责：

- 扫描当前项目目录
- 读取 README、构建文件、配置文件
- 识别入口和核心源码
- 生成本地项目上下文

这个模式适合私有项目、公司项目、不能上传到外部的项目。

## 输出：固定四件事

### 1. 项目地图

回答“这个项目整体是什么”：

- 项目解决什么问题
- 面向什么用户或场景
- 技术栈是什么
- 依赖和资源是什么
- 运行或部署大概需要什么
- 当前扫描到了哪些确定信息
- 哪些信息还缺失

目标：让用户先有全局感，不被目录细节淹没。

### 2. 核心链路

回答“请求、数据或任务是怎么走的”：

- 入口在哪里
- 关键流程经过哪些类或文件
- Controller / Service / Gateway / Scanner 等角色怎么协作
- 如果要改功能，应该从哪里切入
- 每一步对应哪些文件证据

目标：让用户抓住主线，而不是散读文件。

### 3. Top 5 文件

回答“最值得先看的五个文件是什么”：

- 文件路径
- 文件角色
- 为什么重要
- 建议阅读顺序
- 和核心链路的关系

目标：降低第一步阅读成本。

### 4. 阅读报告

回答“用户接下来怎么行动”：

- 最短阅读路径
- 最短 demo 路径
- 需要补的技术点
- 当前风险和不确定点
- 下一步建设建议
- 可导出的 Markdown 总结

目标：从“看懂”推进到“能跑、能改、能讲”。

## 速度要求

产品不能让用户等很久才看到结果。

第一版采用分阶段反馈：

```text
先快速返回项目地图草稿
-> 再补核心链路
-> 再补 Top 5 文件
-> 最后生成完整阅读报告
```

这样即使深度分析还在继续，用户也能先看到主线。

## 缓存要求

缓存是核心能力，不是锦上添花。

需要缓存：

- 项目扫描结果
- README 和构建文件摘要
- 目录树摘要
- Top 5 文件选择结果
- 已生成的项目地图
- 用户历史问题和回答

缓存的价值：

- 同一个项目反复问，不需要重复扫描
- 用户持续追问时，可以复用上下文
- 减少模型调用成本
- 提升响应速度

## 当前最小版本边界

第一阶段先做本地项目模式。

原因：

- 不依赖 GitHub 鉴权
- 不依赖网络
- 更容易验证扫描、结构化输出和缓存设计
- 更适合先用当前项目自举

第一阶段必须做到：

- 输入本地项目路径
- 自动扫描项目
- 输出项目地图
- 输出核心链路
- 输出 Top 5 文件
- 输出阅读报告
- 结果带文件证据

GitHub URL 模式放到下一阶段。

## 作品集亮点

这个项目最终要展示的不是“我调用了 AI API”，而是：

> 我把代码库阅读这件事产品化、流程化、证据化，并用 Spring AI 把模型调用、结构化输出和后续 tool calling 纳入后端工程体系。

可以对外讲：

- 我设计了低门槛输入，只需要 GitHub URL 或本地项目路径。
- 我设计了固定输出结构：项目地图、核心链路、Top 5 文件、阅读报告。
- 我用扫描器自动收集项目上下文，避免用户手工喂材料。
- 我要求结果绑定文件证据，降低幻觉。
- 我通过缓存减少重复扫描和重复模型调用。
- 我用 Spring AI 接入真实模型，并保留 mock 模式保证本地可测试。
