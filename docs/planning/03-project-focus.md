# 项目重点 v1

日期：2026-07-19

## 当前最重要的判断

这个项目的价值不在于“能调用 AI”，而在于把工程阅读做成低门槛、可验证、可持续深挖的工作流。

## 必须守住的 5 个重点

### 1. 低门槛输入

正式产品只接受：

- GitHub URL
- 本地项目路径

不让用户手工整理 README、目录树和配置文件。

### 2. 四件套输出

固定输出：

- 项目地图
- 核心链路
- Top 5 文件
- 阅读报告

不让模型自由发挥成一篇作文。

### 3. 文件证据

每个重要结论尽量指向文件。

如果证据不足，要明确输出 missingContext。

### 4. 快速反馈和缓存

先返回粗结果，再逐步补全。

同一个项目不能每次重新扫描和重新分析。

### 5. 可深挖

第一份报告不是终点。

用户应该能沿着项目地图继续问：

- 某个文件为什么重要
- 某条链路怎么走
- 某个配置为什么生效或不生效
- 某个功能应该从哪里改

## 当前不要做的事

- 不做大而全 agent
- 不做完整 GitHub 平台
- 不做复杂 RAG
- 不做商业化登录注册
- 不做多语言全覆盖
- 不做漂亮但没有后端价值的前端

## 第一阶段最小示例

用当前项目自举：

```text
输入：
/Users/luckyxian/Documents/Codex/2026-07-05/gu/ai-project-reader-learning

输出：
1. 项目地图
2. 核心链路：Controller -> Service -> PromptBuilder -> AiGateway -> Mock/SpringAiGateway

4. 阅读报告
```

这个示例能证明：

- 系统能自动扫描真实项目
- 输出不是裸模型聊天
- 结果有文件证据
- 用户能继续追问 Spring AI 和 AiGateway 细节

## 最值得讲的作品集亮点

一句话：

> 我做了一个基于 Spring AI 的工程热读工作台，它不是简单调用模型，而是自动扫描项目、选择上下文、生成固定结构报告、绑定文件证据，并支持用户继续深挖工程细节。

技术亮点：

- Spring AI `ChatClient`
- `AiGateway` 隔离模型调用
- structured output
- 本地项目扫描
- evidence binding
- caching
- tool calling

产品亮点：

- 低门槛输入
- 快速反馈
- 四件套输出
- 持续追问
- 面向上手和建设
