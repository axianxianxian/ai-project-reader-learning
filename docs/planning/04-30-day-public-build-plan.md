# 30 天公开构建计划

日期：2026-07-19

## 一句话目标

用 30 天做出一个基于 Spring Boot / Spring AI 的工程阅读器：用户给一个 GitHub URL 或本地项目路径，系统能快速生成项目概览、运行方式、核心链路、关键文件，并支持基于项目上下文继续追问。

## 项目要证明什么

这个项目要证明的不是“我会调模型接口”。

它要证明：

> 一个熟后端的人，能把 AI 能力接进真实后端系统，让 AI 按固定流程、工具、缓存、结构化输出和证据规则去阅读一个陌生工程。

裸模型的问题：

- 输入容易不完整
- 输出容易发散
- 抓不到重点
- 没有固定结构
- 没有证据文件
- 每次都像重新问一遍

工程阅读器要解决的是：

- 先扫描项目，而不是直接把问题扔给模型
- 给 AI 固定阅读路径和规则
- 让输出结构化
- 让回答回指文件证据
- 用缓存减少重复分析
- 让小白和容易看深看乱的人先抓主线

## 目标用户

第一类用户是我自己：

- 看到陌生项目容易乱
- 容易一上来读太深
- 容易被资料淹没
- 需要快速抓主线、跑 demo、知道下一步读什么

第二类用户是技术小白或新接手项目的人：

- 不知道 README 该怎么看
- 不知道项目怎么启动
- 不知道哪些文件最重要
- 不知道一个请求或数据链路怎么走

## 输入和输出

### 输入

第一阶段只支持两类输入：

1. 本地项目路径
2. GitHub 仓库链接

当前先做本地项目路径，因为它最容易自举验证。

### 输出

第一阶段固定输出四件套：

1. 项目概览：这个项目解决什么问题，整体结构是什么。
2. 怎么跑起来：依赖、启动方式、最小 demo 路径。
3. 核心链路：入口到核心逻辑怎么走。
4. 关键文件：最值得先读的 5 个文件，以及为什么。

后续再补：

- 风险提示
- 追问问答
- 缓存命中
- 证据文件
- 导出 Markdown

## 为什么用 Spring Boot / Spring AI

我本来就有后端基础，所以不需要从“怎么调 HTTP 接口”开始。

Spring Boot 适合作为产品后端骨架：

- Controller 接收用户输入
- Service 编排扫描和 AI 调用
- Scanner 读取项目上下文
- AiGateway 隔离 mock 和真实模型
- Model 承载结构化输出

Spring AI 适合作为 AI 工程主线：

- 把模型调用纳入 Spring 系统
- 支持结构化输出
- 支持 tool calling
- 后续能接 RAG、memory、observability

## 30 天节奏

工作日最低 45 分钟，理想 1-2 小时。  
周末可以 4-6 小时，用来做大块功能和复盘。

每天必须有一个可见产出：

- 一个页面变化
- 一个接口变化
- 一个结构化对象
- 一篇公开记录
- 一个可运行样例

不要用“我看了很多资料”当产出。

## 今天的小红书文案

标题：

> 第 1 天：我想做一个 AI 工程阅读器

正文：

我最近发现自己学 AI 很容易陷入一个问题：资料越看越多，但真正能拿出来讲的东西很少。

所以我决定做一个 30 天项目：AI 工程阅读器。

它要解决的问题很简单：

当我拿到一个陌生项目时，我希望 AI 不只是随便总结一下，而是能按固定流程帮我读：

1. 这个项目是做什么的
2. 怎么最快跑起来
3. 核心链路怎么走
4. 最值得先读的 5 个文件是什么
5. 如果我继续追问，它能不能带着证据文件回答

我不想只做一个“调模型接口”的 demo。

我想做的是一个真正的 AI 后端工程：

- Spring Boot 做后端
- Spring AI 接模型
- 本地扫描器读取项目上下文
- 输出结构化报告
- 回答必须带证据文件
- 后续加入缓存和持续问答

我的目标不是成为算法工程师，而是成为能把 AI 能力接进真实业务系统的人。

今天先把目标写清楚，也把 demo 页面跑起来。

30 天后，希望它至少能稳定读懂一个我工作里会遇到的工程项目。

今天的问题：

如果你第一次接手一个陌生项目，你最希望 AI 先告诉你什么？

## 今天的 X / Twitter 文案

Day 1 of building an AI project reader.

The goal is not to build another chatbot.

I want to build a Spring Boot / Spring AI app that can read an unfamiliar codebase with a fixed engineering workflow:

- scan the project
- identify the main purpose
- explain how to run it
- trace the core flow
- pick the top 5 files to read first
- answer follow-up questions with file evidence

Bare model calls are too loose for this. Inputs are incomplete, outputs drift, and the model may miss the main path.

So I am turning codebase reading into a structured backend workflow:

Controller -> Service -> Scanner -> Prompt Builder -> AI Gateway -> Structured Report

Day 1 goal: make the product direction clear and get a local demo page running.
