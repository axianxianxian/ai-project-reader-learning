# 风险表 v1

日期：2026-07-19

## 风险 1：项目变成普通 AI 聊天壳

表现：

- 只是把用户问题转给模型
- 没有自动扫描
- 没有固定输出结构
- 没有文件证据

影响：

- 和裸模型没有差异
- 作品集价值弱

应对：

- 固定四件套输出
- 强制 evidenceFiles
- 每个功能都问：它比裸模型多做了什么

## 风险 2：范围膨胀

表现：

- 同时做 GitHub URL、本地扫描、前端、RAG、tool calling、缓存
- 每天都在开新坑

影响：

- 两周内没有可展示成果

应对：

- 第一周只做本地项目四件套
- GitHub URL 放第二阶段
- RAG 放一个月后

## 风险 3：模型输出不稳定

表现：

- 字段缺失
- 内容泛泛
- 文件证据乱写

影响：

- 页面消费困难
- 用户不信任结果

应对：

- structured output
- 输出字段完整性测试
- 缺失上下文字段
- 固定样例评估

## 风险 4：扫描太慢

表现：

- 大项目卡住
- 用户等不到反馈

影响：

- 产品体验差

应对：

- 先扫描 README、构建文件和浅层目录
- 分阶段反馈
- 缓存扫描结果
- 大文件截断

## 风险 5：深究能力不足

表现：

- 用户问细节时只能泛泛回答
- 无法继续读更多文件

影响：

- 产品只能做一次性报告

应对：

- 设计 missingContext
- 设计 nextReadAction
- 后续把 scanner 做成 tool
- 支持文件级和方法级追问

## 风险 6：Spring AI 学成调接口

表现：

- 只会 ChatClient
- 不理解 structured output、tool calling、工程边界

影响：

- 学习价值不够

应对：

- 保留 AiGateway 分层
- 做 tool calling
- 做 mock / spring-ai 切换
- 记录为什么这样设计

## 风险 7：用户自己被信息淹没

表现：

- 每天看很多资料
- 不知道当天产出是什么
- 焦虑后逃避

影响：

- 项目推进中断

应对：

- 每天只做一个小闭环
- 每天有明确验收
- 文档短入口 + 完整资料保留
- 先交付，再深入

## 风险 8：没有真实样例

表现：

- 功能看起来有，但没有实际项目输出

影响：

- 面试和作品集无法说明价值

应对：

- 当前项目自举
- 至少保留一份真实输入输出样例
- 页面展示样例结果

## 风险 9：缓存设计过早复杂

表现：

- 一开始就设计数据库、索引、失效策略

影响：

- 进度变慢

应对：

- 第一版用内存或文件缓存
- 只缓存 ProjectContext 和四件套结果
- 后面再升级

## 风险 10：GitHub URL 依赖外部条件

表现：

- 网络、鉴权、私有仓库权限卡住

影响：

- 第一阶段被外部依赖拖慢

应对：

- 第一阶段先做本地项目
- GitHub URL 放第二阶段
- 先支持公开仓库
